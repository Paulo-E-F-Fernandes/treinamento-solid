# Princípio da Responsabilidade Única (S - Single Responsibility Principle)

## Definição do Princípio da Responsabilidade Única

O princípio da responsabilidade única diz que: **"Cada classe deve ter um, e somente um, motivo para mudar"**.

Se uma classe tem várias responsabilidades, mudar um requisito do projeto pode trazer várias razões para modificar a
classe. Por isso, as classes devem ter responsabilidades únicas.

Esse princípio pode ser estendido para os métodos que criamos também. Quanto mais tarefas um método executa, mais difícil
é testá-lo e garantir que o programa está em ordem.

Uma dica para aplicar esse princípio, é tentar nomear as classes ou métodos com tudo que eles são capazes de fazer.

Se o nome ficar gigante, como `GerenciadorTarefasEmailsRelatorios`, temos um sinal de que o código deve ser refatorado.

## Vantagens de aplicar o Princípio da Responsabilidade Única

Existem vários benefícios ao aplicar esse princípio, principalmente:

- Facilidade para fazer manutenções
- Reusabilidade das classes
- Facilidade para realizar testes
- Simplificação da legibilidade do código

Para entender o princípio da responsabilidade única, vamos pensar no desenvolvimento de um gerenciador de tarefas.

Vamos começar com o código da classe
`br/com/paulofernandes/solid/singleresponsibilityprinciple/problem/GerenciadorTarefas.java`.

## Problemática

Vamos enumerar todas as funções que a classe `GerenciadorTarefas` possui. Essa classe é responsável por lidar com todas
as operações das tarefas, mas também está consumindo uma _API_, enviando notificações para usuários, gerando e 
enviado os relatórios gerados pela aplicação.

Pensando em Orientação a Objetos, um objeto **gerenciador de tarefas** deveria enviar os e-mails e também gerar 
relatórios?
- **Não!**

Um gerenciador de tarefas deve gerenciar as tarefas da aplicação, e não e-mails ou gerar e enviar relatórios.

Verificando a classe `GerenciadorTarefas` do pacote `.../singleresponsibilityprinciple/problem/`, não é legal que uma
única classe tenha 1001 utilidades, pois pode gerar confusão no entendimento da funcionalidade que é realizada, e 
dificuldade de manter e evoluir a classe.

O ideal é que tenhamos várias classes diferentes, e que cada uma delas tenha a própria responsabilidade executando uma
função específica.

## Solução

Para resolver esse problema, vamos criar classes diferentes, sendo que cada uma dessas novas classes vai representar uma 
função diferente.

A nossa classe `GerenciadorTarefas` vai ter apenas os códigos relacionados as operações com as tarefas. As outras
operações estarão em outras classes. 

Dessa forma, cada classe será responsável por uma parte diferente da aplicação. Assim, teremos a classe `GerenciadorTarefas` 
refatorada, e por motivos didáticos, vamos criar um novo pacote chamado
`.../singleresponsibilityprinciple/solution/`
para termos o exemplo da classe de problema e da classe refatorada com a solução. 

Dessa forma, a classe `GerenciadorTarefas` do pacote `.../singleresponsibilityprinciple/solution/`, representa a 
classe refatorada.

- `br/com/paulofernandes/solid/singleresponsibilityprinciple/solution/GerenciadorTarefas.java`

Também criamos uma classe específica para consumir uma _API_ externa, uma classe para enviar as notificações e uma 
última para gerar os relatórios.

- `br/com/paulofernandes/solid/singleresponsibilityprinciple/solution/ConectarAPI.java`
- `br/com/paulofernandes/solid/singleresponsibilityprinciple/solution/Notificador.java`
- `br/com/paulofernandes/solid/singleresponsibilityprinciple/solution/GeradorRelatorio.java`

Tá, mas as classes não ficaram pequenas demais. Nesse caso, não estão pequenas, pois cada classe reflete exatamente a 
responsabilidade que a classe possui.

Então, dessa forma, se for necessário adicionar um novo método relacionado ao consumo da _API_, sabemos exatamente em 
qual classe o código deve ser escrito, ficando muito mais simples alterar o que for preciso.
